# Incident Postmortem: NorthBridge Freight Routing Outage — March 14, 2024

## Executive Summary
On March 14, 2024, NorthBridge Logistics experienced a 4-hour 17-minute service degradation affecting real-time route optimization and dispatch coordination across the Mid-Atlantic hub. The incident was triggered by a corrupted geospatial cache update deployed at 06:42 EST, resulting in 1,247 delayed shipments and an estimated revenue impact of $89,000.

## Timeline of Events
- **06:42 EST** — Automated deployment of v2.14.3 geospatial patch to production cluster.
- **06:51 EST** — Monitoring alerted on elevated 502 errors from the routing API; on-call engineer Marcus Chen acknowledged.
- **07:15 EST** — Incident declared Severity-1; customer-facing tracking updates suspended.
- **08:30 EST** — Rollback to v2.14.2 completed, but corrupted cache persisted in Redis nodes.
- **10:59 EST** — Full service restoration confirmed after manual cache purge and validation.

## Root Cause Analysis
The deployment script failed to validate the spatial index checksum before writing to the primary Redis cluster. When the routing engine queried boundary polygons for zip codes in the 21201–21230 range, it returned null coordinates, causing dispatcher maps to render blank tiles and forcing manual fallback procedures.

## Impact Assessment
- 1,247 shipments delayed by an average of 3.2 hours
- 34 commercial clients affected, including Meridian Grocers and Atlantic Cold Storage
- Customer support volume spiked 340% between 07:00 and 11:00 EST
- Zero data loss or security breaches detected

## Remediation Actions
- Implement checksum validation gates in CI/CD pipeline for all geospatial assets
- Add redundant read-only cache replicas in the Baltimore and Richmond regions
- Update runbooks to include explicit Redis purge commands for corrupted spatial indices
- Conduct quarterly chaos-engineering exercises on the dispatch grid

## Lessons Learned
The outage revealed gaps in deployment guardrails and cross-team communication. Marcus Chen’s rapid acknowledgment prevented wider cascade failure, but the absence of an automated rollback for cache layers extended recovery time. Going forward, all routing data updates require dual-engineer approval and staged rollout to 5% of nodes before full promotion.
