# VaultPay Technical Design Document

## Overview
VaultPay is a real-time peer-to-peer settlement platform targeting small-business treasury workflows. The product must support instant ACH reversals, multi-currency ledgers, and sub-100ms latency for 99.99% of requests. The initial build, funded by a $2.4M seed round led by Meridian Capital, targets a closed beta of 500 merchants across the U.S. and Canada.

## System Architecture
The backend runs on a Kubernetes cluster in three AWS regions with active-active replication. Core services include Ledger Engine (Go), Settlement Router (Rust), and Notification Dispatcher (Node). Data flows through an event bus (Apache Kafka) with exactly-once semantics. The mobile clients are native Swift (iOS) and Kotlin (Android), communicating over gRPC-Web secured with mTLS.

- Ledger Engine handles double-entry bookkeeping with idempotency keys
- Settlement Router interfaces with FedNow and RTP networks via adapter plugins
- Notification Dispatcher queues SMS, push, and email via SendGrid
- Observability stack uses Prometheus, Grafana, and Jaeger distributed tracing

## Security & Compliance
All sensitive data is encrypted at rest using AES-256-GCM and in transit via TLS 1.3. The platform undergoes quarterly SOC 2 Type II audits and maintains PCI DSS Level 1 certification. Access control enforces least-privilege through Okta SSO with hardware-key MFA for all engineers. Personal Identifiable Information (PII) is tokenized before reaching analytics warehouses.

## Implementation Timeline
Phase 1 (Core Ledger) runs from June 1, 2024 to August 15, 2024. Phase 2 (Payment Rails Integration) begins August 16, 2024 and closes October 31, 2024. The public launch is scheduled for January 14, 2025, contingent on regulatory clearance from the OCC and FINTRAC. Engineering headcount will scale from 8 to 22 by Q4.

## Risk Mitigation
Key risks include settlement failure under peak load, regulatory delays in Canada, and dependency on FedNow uptime. We maintain a 72-hour rollback window for every deployment and run chaos tests every Tuesday at 02:00 UTC. A dedicated SRE rotation ensures 24/7 incident response with a 15-minute escalation path to the CTO.
