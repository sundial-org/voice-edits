# Incident Postmortem: Learnhub Platform Outage — March 14, 2024

## Executive Summary
On March 14, 2024, the Learnhub learning management system experienced a 42-minute total service outage between 08:47 UTC and 09:29 UTC. The incident affected approximately 12,400 students and 860 educators across 14 partner districts. The root cause was an untested database migration script deployed during the nightly maintenance window by engineer Marcus Chen, which triggered a cascading lock failure on the primary PostgreSQL cluster hosted in us-east-1.

## Timeline
- **08:17 UTC** — Maintenance window begins; migration script v2.4.1 initiated.
- **08:47 UTC** — Monitoring alert fired: database connection pool exhausted.
- **08:52 UTC** — Dr. Aisha Patel, Director of Engineering, paged and acknowledged incident.
- **09:05 UTC** — Failover to read replica attempted; replication lag exceeded 18 minutes.
- **09:29 UTC** — Script rolled back, connections drained, and full service restored.
- **10:15 UTC** — All cached session data rebuilt; status page updated.

## Root Cause Analysis
The migration script altered the `enrollment_logs` table index without acquiring an advisory lock. Under concurrent write load from early-morning attendance uploads, this produced dead tuples faster than autovacuum could reclaim them. Connection pool saturation followed within four minutes. The script had passed unit tests but was never executed against the production-sized dataset of 3.2 million rows.

## Impact Assessment
- **12,400 active students** lost access to live coursework and assignment submissions during the outage window.
- **860 educators** in 14 districts could not record attendance or view real-time progress dashboards.
- **3 partner schools** in the Denver pilot program missed synchronized assessment windows and required manual rescheduling.
- Zero data corruption or unauthorized access occurred; all transactional logs were preserved.
- Estimated volunteer tutor hours disrupted: approximately 210 scheduled sessions.

## Remediation Plan
We will require peer review for all schema changes, implement automated canary testing against a 10% traffic shadow cluster, and extend the maintenance blackout window to exclude 08:00–09:30 UTC in partner time zones. Marcus Chen and the platform team will complete a runbook update by March 28, 2024.

## Lessons Learned
Outage duration exceeded our 15-minute SLO because failover procedures assumed replication lag under five minutes. Going forward, any migration touching tables larger than 1 million rows must run during low-traffic weekends with explicit freeze protocols. Dr. Aisha Patel will present revised change-management policies at the board meeting on April 3, 2024.
